=====================================================
B) STRUCTURAL DESIGN PATTERNS
=====================================================

1) What are Structural Design Patterns?
--------------------------------------
Structural Design Patterns deal with HOW classes and
objects are COMPOSED to form larger structures.

They focus on:
- Class relationships
- Object composition
- Simplifying complex structures

In simple words:
→ They define how objects are connected and organized


-----------------------------------------------------

2) Problem Without Structural Design Patterns
----------------------------------------------
Without structural patterns:

❌ Tight coupling between classes
❌ Complex object relationships
❌ Difficult to extend existing code
❌ Poor readability
❌ Hard to maintain large systems

Example:
Service directly depends on multiple concrete classes


-----------------------------------------------------

3) Solution by Using Structural Design Patterns
-----------------------------------------------

| Problem                         | Solution Provided                |
|---------------------------------|----------------------------------|
| Tight coupling                  | Use abstraction & composition    |
| Complex object relationships    | Wrap & simplify structure        |
| Incompatible interfaces         | Use adapters                     |
| Feature explosion via subclass  | Use decorators                   |
| Heavy objects                   | Share objects efficiently        |


-----------------------------------------------------

4) Key Focus of Structural Patterns
-----------------------------------
- How objects are CONNECTED
- How classes are COMPOSED
- How structure can be EXTENDED
- How to reuse existing code safely


-----------------------------------------------------

5) Types of Structural Design Patterns
--------------------------------------

| Pattern       | Purpose                                         |
|--------------|--------------------------------------------------|
| **Adapter**  | Convert one interface into another               |
| **Bridge**   | Separate abstraction from implementation         |
| **Composite**| Treat individual & group objects uniformly       |
| **Decorator**| Add behavior dynamically                         |
| **Facade**   | Provide a simplified interface                  |
| **Flyweight**| Reduce memory by sharing objects                 |
| **Proxy**    | Control access to another object                 |


-----------------------------------------------------

6) One-Line Explanation of Each Pattern
---------------------------------------

Adapter:
→ Makes incompatible interfaces work together

Bridge:
→ Decouples abstraction from implementation

Composite:
→ Treats individual and group objects the same

Decorator:
→ Adds behavior without modifying class

Facade:
→ Simplifies complex subsystems

Flyweight:
→ Reuses objects to save memory

Proxy:
→ Controls access to real object


-----------------------------------------------------

7) Where are Structural Patterns Used (Real World)
--------------------------------------------------
- Spring Framework (Proxy, Decorator)
- Logging frameworks
- UI frameworks
- ORM frameworks (Hibernate)
- Security & access control
- Performance optimization


-----------------------------------------------------

8) Structural vs Creational (Quick Comparison)
-----------------------------------------------

Creational Patterns:
- Focus on object creation
- Decide HOW objects are created

Structural Patterns:
- Focus on object composition
- Decide HOW objects are connected

=====================================================
ADAPTER DESIGN PATTERN
=====================================================

1) What is Adapter Pattern?
--------------------------
Adapter pattern is a structural design pattern that
allows TWO INCOMPATIBLE interfaces to work together.

It acts as a bridge between:
- Existing class (old / third-party)
- New system expecting a different interface

In simple words:
→ Adapter converts one interface into another


-----------------------------------------------------

2) Why Do We Need Adapter?
-------------------------
Problem:
Existing class interface does NOT match what
client code expects.

Example:
Client expects:
    pay(int amount)

But existing class provides:
    makePayment(double money)


-----------------------------------------------------

3) Real-Life Example
--------------------
- Power adapter (2-pin → 3-pin)
- Card reader for laptop
- USB-C to HDMI adapter


-----------------------------------------------------

4) How do we know it is Adapter Pattern?
---------------------------------------
- Client expects a TARGET interface
- Existing class has a DIFFERENT interface
- Adapter implements TARGET interface
- Adapter internally uses existing class

Key Identifier:
→ Wrapper class that translates method calls


-----------------------------------------------------

5) Structure of Adapter Pattern
--------------------------------
Target
  └── Interface expected by client

Adapter
  └── Implements Target
  └── Wraps Adaptee

Adaptee
  └── Existing / incompatible class

Client
  └── Uses Target interface


-----------------------------------------------------

6) What Problem Does It Solve?
------------------------------
Problems:
❌ Incompatible interfaces
❌ Cannot modify third-party code
❌ Tight coupling if modified directly

Solutions:
✅ Makes incompatible classes work together
✅ Reuses existing code
✅ No changes to original class


-----------------------------------------------------

7) Types of Adapter Pattern
---------------------------
1) Class Adapter
   - Uses inheritance
   - Less common in Java

2) Object Adapter (MOST COMMON)
   - Uses composition
   - Preferred in Java


-----------------------------------------------------

8) Advantages
-------------
✔ Enables reuse of existing code
✔ Promotes loose coupling
✔ Improves maintainability


-----------------------------------------------------

9) Disadvantages
----------------
❌ Extra layer of abstraction
❌ Slightly more complex code


-----------------------------------------------------

10) Real-World Usage
--------------------
- Legacy system integration
- Third-party libraries
- Payment gateway integration
- Spring MVC HandlerAdapter


-----------------------------------------------------

11) Adapter vs Facade (IMPORTANT)
---------------------------------
Adapter:
- Makes incompatible interfaces compatible
- Works on existing interface

Facade:
- Simplifies complex subsystem
- Provides new interface

-----------------------------------------------------

12) One-Line Interview Answer
----------------------------
Adapter pattern allows incompatible interfaces to
work together by converting one interface into
another expected by the client.

DECORATOR DESIGN PATTERN
=====================================================

1) What is Decorator Pattern?
----------------------------
Decorator pattern is a structural design pattern that
allows behavior to be ADDED to an object dynamically
without modifying its existing class.

In simple words:
→ Add new features without changing the original code


-----------------------------------------------------

2) Why Do We Need Decorator?
---------------------------
Problems without Decorator:
❌ Too many subclasses for each feature
❌ Inheritance explosion
❌ Hard to add features dynamically

Example:
Coffee
- CoffeeWithMilk
- CoffeeWithMilkAndSugar
- CoffeeWithMilkSugarAndCream


-----------------------------------------------------

3) How do we know it is Decorator Pattern?
-----------------------------------------
- A COMPONENT interface exists
- Concrete component implements it
- Decorator implements SAME interface
- Decorator WRAPS another component

Key Identifier:
→ Wrapper class that adds behavior


-----------------------------------------------------

4) Structure of Decorator Pattern
---------------------------------
Component
  └── Interface for objects

ConcreteComponent
  └── Original object

Decorator
  └── Implements Component
  └── Has reference to Component

ConcreteDecorator
  └── Adds new behavior


-----------------------------------------------------

5) What Problem Does It Solve?
------------------------------
Problems:
- Feature explosion using inheritance
- Inflexible object behavior
- Hard to combine features

Solutions:
- Add behavior at runtime
- Flexible feature combinations
- No modification of existing classes


-----------------------------------------------------

6) Advantages
-------------
✔ Open–Closed Principle
✔ Flexible and dynamic
✔ Avoids subclass explosion


-----------------------------------------------------

7) Disadvantages
----------------
❌ Many small classes
❌ Can be harder to debug
❌ Order of decorators matters


-----------------------------------------------------

8) Decorator vs Inheritance
---------------------------
Decorator:
- Adds behavior dynamically
- More flexible

Inheritance:
- Adds behavior at compile time
- Less flexible


-----------------------------------------------------

9) Real-World Usage
-------------------
- Java I/O streams
- Spring AOP
- Servlet Filters
- Logging frameworks


-----------------------------------------------------

10) One-Line Interview Answer
----------------------------
Decorator pattern adds new behavior to an object
dynamically without modifying its class.


-----------------------------------------------------

11) Ultra-Short Revision
-----------------------
- Wrapper pattern
- Dynamic behavior
- No class modification
- Alternative to inheritance

=====================================================
FACADE DESIGN PATTERN
=====================================================

1) What is Facade Pattern?
-------------------------
Facade pattern is a structural design pattern that
provides a SIMPLIFIED interface to a complex
subsystem.

In simple words:
→ One simple interface instead of many complex ones


-----------------------------------------------------

2) Why Do We Need Facade?
------------------------
Problems without Facade:
❌ Client must know many classes
❌ Complex subsystem interactions
❌ Tight coupling with subsystem

Example:
Starting a computer:
- CPU start
- Memory load
- Disk read
Client should not handle all these


-----------------------------------------------------

3) How do we know it is Facade Pattern?
--------------------------------------
- Multiple complex subsystem classes exist
- A single class provides simplified methods
- Client interacts only with the Facade
- Facade delegates work to subsystems

Key Identifier:
→ One entry point hiding complexity


-----------------------------------------------------

4) Structure of Facade Pattern
------------------------------
Facade
  └── Simplified interface

Subsystem Classes
  └── Complex logic

Client
  └── Uses only Facade


-----------------------------------------------------

5) What Problem Does It Solve?
------------------------------
Problems:
- Complex API usage
- Tight coupling
- Poor readability

Solutions:
- Simplifies usage
- Decouples client from subsystem
- Improves maintainability


-----------------------------------------------------

6) Advantages
-------------
✔ Simplifies client code
✔ Reduces coupling
✔ Improves readability


-----------------------------------------------------

7) Disadvantages
----------------
❌ Facade can become a god class
❌ May hide useful subsystem features


-----------------------------------------------------

8) Facade vs Adapter (IMPORTANT)
--------------------------------
Facade:
- Simplifies an interface
- Does NOT change existing interfaces

Adapter:
- Converts one interface into another
- Makes incompatible interfaces work


-----------------------------------------------------

9) Real-World Usage
-------------------
- Logging frameworks
- Spring JdbcTemplate
- Hibernate Session
- API gateways


-----------------------------------------------------

10) One-Line Interview Answer
----------------------------
Facade pattern provides a simplified interface to
a complex subsystem, hiding its internal details.


-----------------------------------------------------

11) Ultra-Short Revision
-----------------------
- Simplified interface
- Hide complexity
- One entry point
- Decoupling

=====================================================
PROXY DESIGN PATTERN
=====================================================

1) What is Proxy Pattern?
------------------------
Proxy pattern is a structural design pattern that
provides a PLACEHOLDER or SUBSTITUTE for another
object to CONTROL access to it.

In simple words:
→ Proxy controls access to a real object


-----------------------------------------------------

2) Why Do We Need Proxy?
-----------------------
Problems without Proxy:
❌ Direct access to sensitive objects
❌ Expensive object creation
❌ No access control

Example:
- Loading large image
- Securing service access
- Lazy initialization


-----------------------------------------------------

3) How do we know it is Proxy Pattern?
-------------------------------------
- Proxy and real object implement SAME interface
- Client interacts with Proxy
- Proxy delegates to real object
- Proxy controls access

Key Identifier:
→ Wrapper that controls access


-----------------------------------------------------

4) Structure of Proxy Pattern
------------------------------
Subject
  └── Common interface

RealSubject
  └── Actual object

Proxy
  └── Controls access to RealSubject

Client
  └── Uses Proxy


-----------------------------------------------------

5) Types of Proxy
-----------------
1) Virtual Proxy
   - Lazy initialization

2) Protection Proxy
   - Access control

3) Remote Proxy
   - Represents remote object

4) Cache Proxy
   - Caches results


-----------------------------------------------------

6) What Problem Does It Solve?
------------------------------
Problems:
- Uncontrolled access
- High resource usage
- Tight coupling

Solutions:
- Controlled access
- Lazy loading
- Improved performance


-----------------------------------------------------

7) Advantages
-------------
✔ Controls access
✔ Improves performance
✔ Adds security
✔ Lazy initialization


-----------------------------------------------------

8) Disadvantages
----------------
❌ Extra layer of abstraction
❌ Increased complexity
❌ Slight performance overhead


-----------------------------------------------------

9) Proxy vs Decorator (IMPORTANT)
---------------------------------
Proxy:
- Controls access
- Focus on lifecycle & security

Decorator:
- Adds behavior
- Focus on functionality


-----------------------------------------------------

10) Real-World Usage
--------------------
- Spring AOP proxies
- Hibernate lazy loading
- Security frameworks
- Remote services (RMI)


-----------------------------------------------------

11) One-Line Interview Answer
----------------------------
Proxy pattern provides a surrogate or placeholder
object to control access to another object.


-----------------------------------------------------

12) Ultra-Short Revision
-----------------------
- Access control
- Lazy loading
- Security
- Wrapper object
